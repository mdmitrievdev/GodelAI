# PR Review Assistant — AI Generation Strategy & Copilot Rules

---

## 1. Core Rule: 90%+ AI-Generated Code

Every file must be generated by GitHub Copilot (or another AI tool) unless there is a technical reason it cannot be. This document defines the strategy, workflow, and enforcement rules.

---

## 2. Coding Workflow

### Step-by-Step Process for Each Task

1. **Open a new Copilot chat** (one task = one chat)
2. **Provide context**: reference the SRS section, existing interfaces, and related files
3. **Request a plan first** (Plan mode) — review before implementation
4. **Approve the plan**, then switch to Agent mode for implementation
5. **Review generated code** against the checklist below
6. **Log the prompt** — save as `{IDENTITY_NUMBER}_{SHORT_KEY}.md` in `prompts/` (convention: see `.github/instructions/prompt-logging.instructions.md`, template: `prompts/_prompt-log-template.md`)
7. **Commit** with a Conventional Commits message

### Never Do
- Generate entire project in one prompt
- Mix frontend and backend in the same prompt
- Skip interface generation and jump to implementation
- Accept code without reviewing against conventions
- Manually write code without trying AI first

---

## 4. Prompt Templates

### 4.1 Generate a Domain Entity

```
Generate a C# entity class for {EntityName} in the Domain/Entities folder.
Fields: {list fields with types}.
Use string (ULID) for the Id.
Use DateTime (UTC) for timestamps.
Navigation properties where appropriate.
Follow the conventions in .github/copilot-instructions.md.
```

### 4.2 Generate a DTO

```
Generate a C# record for {DtoName} in the Shared/DTOs folder.
Fields: {list fields with types}.
Use record syntax.
Use IReadOnlyList<T> for collections.
Follow API contract conventions: camelCase JSON, string IDs, ISO 8601 timestamps.
```

### 4.3 Generate a MediatR Handler

```
Generate a MediatR {Command|Query} and Handler for {FeatureName}.
Location: Features/{Domain}/{Commands|Queries}/
The handler should:
  - {describe what it does}
  - Use {IReviewRepository|IAiAnalysisService} via constructor injection
  - Include CancellationToken on all async methods
  - Return {ReturnType}
Follow the MediatR pattern from .github/copilot-instructions.md.
```

### 4.4 Generate a FluentValidation Validator

```
Generate a FluentValidation validator for {RequestType}.
Rules:
  - {field}: {validation rule}
  - {field}: {validation rule}
Location: Features/{Domain}/Validators/
```

### 4.5 Generate an API Endpoint Map

```
Generate a Minimal API endpoint map for {Resource} endpoints.
Endpoints:
  - {METHOD} /api/v1/{path} → Send({CommandOrQuery})
  - ...
Location: Endpoints/{Resource}Endpoints.cs
Each endpoint should only call mediator.Send() — no business logic.
Wrap responses in ApiResponse<T>.
```

### 4.6 Generate a React Component

```
Generate a React TypeScript functional component named {ComponentName}.
Props: {list props with types, define interface above component}.
It should: {describe rendering and behavior}.
Use the API service from services/api.ts.
Error handling: display errors via <ErrorMessage> component.
No any types. Explicit return type.
Location: components/{subfolder}/{ComponentName}.tsx
```

### 4.7 Generate a React Page

```
Generate a React TypeScript page component for {PageName}.
Route: {route path}
Layout: {MainLayout | AdminLayout}
It should: {describe page functionality}.
Use hooks: {list custom hooks to use}.
Compose these components: {list existing components}.
Handle loading and error states.
Location: pages/{PageName}.tsx
```

### 4.8 Generate a Custom Hook

```
Generate a React TypeScript custom hook named {useHookName}.
Purpose: {describe what data it fetches/manages}.
Use the API service from services/api.ts.
Return type: { data: T | null, loading: boolean, error: string | null, ...actions }
Handle errors at the API boundary.
No any types. Explicit return type.
Location: hooks/{useHookName}.ts
```

### 4.9 Refactoring

```
Refactor {file or component name}.
Current issues: {describe problems}.
Goals: {describe desired outcome}.
Keep the same external interface/API.
Do not break existing tests.
```

### 4.10 Adding Tests

```
Generate {unit|integration} tests for {target}.
Location: {test file path}
Use {xUnit|Jest + RTL}.
Test these scenarios:
  - {scenario 1}
  - {scenario 2}
Follow AAA pattern. One assertion per test.
Name format: {Method}_{Scenario}_{ExpectedResult}
```

## 5. Rules for Manual vs AI Code

| Scenario | Who Writes | Documentation |
|----------|-----------|---------------|
| New file from scratch | AI | Save prompt as `{N}_{key}.md` in `prompts/` |
| Bug fix (< 5 lines) | AI or manual | If manual: add `// MANUAL EDIT: <reason>` |
| Configuration tweak | Manual OK | Note in corresponding prompt log file |
| Package.json / csproj edits | AI or manual | Note in corresponding prompt log file |
| Connection strings / env vars | Manual OK | Document in .env.example |
| AI-generated code that doesn't compile | Fix with AI first | If manual fix: mark with comment |

### Escalation Rule
If AI cannot produce correct code after 2 attempts:
1. Simplify the prompt (break into smaller pieces)
2. Provide more context (reference existing files)
3. If still failing: manual fix with `// MANUAL EDIT: AI failed after 2 attempts - <description>`

## 6. Review Checklist (Before Accepting AI Code)

### Universal
- [ ] Code compiles without errors
- [ ] No `console.log`, `debugger`, or TODO left
- [ ] No unused imports or variables
- [ ] File is in the correct location per folder structure
- [ ] Naming follows conventions

### TypeScript / React
- [ ] No `any` types
- [ ] No `as unknown as X`
- [ ] Props interface defined above component
- [ ] Explicit return types on exported functions
- [ ] Error handling present (no empty catch blocks)
- [ ] No raw `console.log` in production paths

### C# / .NET
- [ ] Async methods end with `Async` suffix
- [ ] `CancellationToken` on all async methods
- [ ] Private fields use `_camelCase`
- [ ] No business logic in endpoints (only `Send()`)
- [ ] FluentValidation for input validation
- [ ] Returns ProblemDetails on errors
- [ ] Uses `record` for DTOs
- [ ] Uses `IReadOnlyList<T>` for return collections

## 7. Metrics: Measuring AI-Generated Percentage

### Method
Track each file in a table:

| File | Generated By | Manual Lines | Total Lines | AI % |
|------|-------------|-------------|-------------|------|
| Review.cs | Copilot | 0 | 25 | 100% |
| Program.cs | Copilot + Manual | 3 | 40 | 92.5% |

### Calculation
```
AI% = (Total lines - Manual lines) / Total lines × 100
Project AI% = Sum(AI lines across all files) / Sum(Total lines) × 100
```

### Target: 90%+ overall

## 9. Consistency Enforcement

### Before Starting Each Session
1. Verify `.github/copilot-instructions.md` is present
2. Verify existing code follows established patterns
3. Reference existing files when generating new code

### When Generating Related Files
- Always reference the interface when generating implementation
- Always reference the entity when generating DTOs
- Always reference the DTO when generating frontend models
- Always reference existing components when generating pages

### Pattern Lock-In
Once a pattern is established in the first file of a kind (e.g., first Controller, first Component), all subsequent files of the same kind must follow the exact same pattern.

## 10. Validation Before Commit

Run before every commit:

```bash
# Backend
cd backend
dotnet build
dotnet test

# Frontend
cd frontend
npm run build
npm test -- --passWithNoTests
npm run lint
```

All must pass. Fix failures before committing.
